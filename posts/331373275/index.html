AI 에이전트를 위한 새로운 CLI 설계 원칙

이전 글에서 얘기한 것처럼, AI 에이전트를 위한 도구는 CLI가 제일 적합하다. 그런데 여기에서 그치지 않고 CLI를 어떻게 작성해야 하는지에 대한 좋은 내용이 있어서 소개한다.

사람을 위한 CLI와 AI 에이전트를 위한 CLI는 설계 목표부터가 완전히 다르다. 지금까지 우리가 만든 도구들은 사람이 읽기 편하고 실수해도 너그럽게 받아주는 데 최적화되어 있었지만, 에이전트가 주 사용자가 되는 시대에는 예측 가능성과 보안이 최우선 순위가 되어야 한다.

기존의 CLI를 에이전트용으로 단순히 개조하는 것은 비효율적이다. 사람은 터미널에서 명령어를 하나씩 입력하며 도움말을 찾아보지만, 에이전트는 한 번에 완벽한 구조의 데이터를 전달하고 결과가 기계가 읽기 쉬운 형태이기를 원한다. 따라서 에이전트 전용 CLI는 발견 가능성보다는 다층 방어와 예측 가능성에 집중해야 한다.

구체적으로는 개별 플래그를 나열하는 방식보다 전체 API 페이로드를 JSON으로 전달하는 방식이 에이전트에게 훨씬 유리하다. 복잡한 중첩 구조를 표현하기에도 좋고 LLM이 생성하기에도 직관적이기 때문이다. 또한 CLI 자체가 런타임에서 질의 가능한 스키마 정보를 제공해야 한다. 에이전트가 문서를 검색하느라 토큰을 낭비하지 않도록, 명령어를 통해 현재 API의 사양을 즉시 파악하게 만드는 것이 핵심이다. CLI 자체가 문서의 역할을 수행해야 에이전트의 효율이 극대화된다.

에이전트는 신뢰할 수 있는 운영자가 아니라는 점을 명심해야 한다. 에이전트는 할루시네이션으로 인해 존재하지 않는 경로를 입력하거나 이상한 제어 문자를 포함하는 실수를 저지를 수 있다. 웹 API에서 사용자 입력을 검증하듯, CLI 수준에서도 파일 경로 샌드박싱이나 특수 문자 차단 같은 강력한 입력 검증 장치를 마련해야 한다. 에러가 발생했을 때 심각한 데이터 손실을 막기 위해 실제 실행 전 요청을 로컬에서 검증하는 드라이 런(dry-run) 기능도 반드시 포함해야 한다.

에이전트의 컨텍스트 윈도우를 효율적으로 관리하는 것도 CLI의 중요한 임무다. API 응답 중 불필요한 필드는 에이전트의 추론 능력을 저하시키고 비용을 발생시킨다. 따라서 필요한 필드만 골라내는 필드 마스크 기능을 제공하여 응답 크기를 제한해야 한다. 또한 에이전트에게는 인간과 같은 직관이 없으므로, 어떤 상황에서 어떤 명령을 써야 하는지 명시한 스킬 파일을 제공하는 것이 효과적이다. 이는 시스템 프롬프트에 정적인 문서를 넣는 것보다 훨씬 유연하고 비용 효율적인 방식이다.

에이전트 퍼스트 설계는 단순한 편의 기능이 아니라 AI 시대에 살아남기 위한 새로운 표준이 될 것이다.
